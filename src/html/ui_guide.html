UI Guide

🐌 공통

  • - 기본적인 웹 표준, 접근성, SEO에 대한 개념을 숙지하고 있어야 합니다.
  • - 서비스별 코딩 규약과 특징을 정확히 구분하고 있어야 합니다.
  • - 기본 가이드를 벗어나는 코딩 방식을 선택할 때는 반드시 사전 협의가 있어야 합니다.
  • - 플러그인은 협의된 기본적인 것만 사용 추가로 사용시 반드시 사전 협의가 있어야합니다.
  • - HTML5, CSS, SCSS, svg, gulp, jquery-1.12.4(IE8 지원) 기준
  • - slick.js, Iscroll.js (일부 프로젝트는 bx슬라이드를 쓰지만 버그가 많아 slick 사용)
  • - 신규 구축 시 UI 템플릿 사용
  • - style 기능별 모듈화 필수 (페이지별 모듈화는 사용하지 않는다)
  • - css해결 가능한건 css에서 해결, js는 꼭 필요한 경우에 적용

🐌 Naming

  • - 이름 맨 앞에 특수문자, 숫자는 사용안됨 (마크업 폴더 예외)

    ex) 1img_noti (x), img_noti (o)

  • - 넘버링은 이름 다름 바로

    ex) img_noti_02 (x), img_noti1 (o)

  • - 형태_ 의미_상태 3단계를 넘어가지 않게 작성

🐌 HTML - gulp 기준

  • - 시멘틱 태그 사용

    웹접근성, 웹표준, seo 준수

  • - 비슷한 형태는 따로 파일을 만들지 않고 gulp 기능을 이용하여 처리

    ex) header_main(x), tit = "회사소개" @@tit (o)

  • - 다른 팀원과 같이 작업이 가능하게 모듈페이지 작성
  • - 다국어페이지 같은 경우 1가지 마크업으로 적용(서비스하는 모든 언어에서 작업할수있게 작업)

      ├── header.html
      ├── footer.html
      ├── location.html
      ├── paging.html
      └── pop

🐌 스타일

  • - 전처리 도구는 scss를 사용
  • - autoprefixer는 gulp 빌드 시 자동으로 추가

    souremap이 안맞아서, min 폴더 안에 있는 css에만 적용

  • - reset에는 페이지 구성에 가장 기본적인 것만 넣는다.(모든 프로젝트 공통으로 사용 가능)
  • - 이미지 버전 관리는 ...?v=날짜

    실서버 적용 후 사용안하는 이미지는 삭제 해야되는데, 자주 누락 되서 캐싱 처리 방법으로 변경

  • - id는 반드시 유니크 해야한다.
  • - id는 퀵네비 또는 페이지내 이동 시를 제외하고 사용하지 않는다.
  • - id는 camelcase 방식으로 하며, class는 underscore 방식으로 사용한다.
  • - 헷갈릴 소제가 있는 이름 경우 절대 단독클레스로 사용하지 않는다(전역사용 x, 중복클레스사용 o)

    ex) left, right, top, bottom, big, box 등 (x) .btn.big, cont.left(o)

  • - class 이름은 기능을 알수있게 명확하게 표시

    ex) tbl1 (x), `tbl board(o)`

  • - 공통요소는 common.ccss

    ex) common에 있는것들은 예외는 되도록 만들지 않는다.

  • - 공통으로 사용 되는 스타일 코드는 mixin.scss의 function, mixin 참고
  • - 임시 이미지는 temp_*.* 사용 후 실서비스 적용 후 삭제
